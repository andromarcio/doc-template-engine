# N2 — Feature Set

Um **conjunto de funcionalidades** relacionadas dentro de um domínio (ex.: dentro
de Vendas → Carrinho, Checkout).

## O papel do N2

É a **ponte** entre a visão de domínio (N1) e a especificação fina de cada feature
(N3). Seu ID `[SIGLA]-[SFS]` (ex.: `USR-PRM`) é **definido já no N1**, ao listar os
Feature Sets do domínio — o N2 apenas o reutiliza.

## O que contém

- **Jornada principal do usuário** — um diagrama de fluxo **só para frente** (sem caminho de volta; retornos e exceções ficam no N3).
- **Features** que compõem o conjunto.
- **Telas** envolvidas.
- **Permissões por perfil** — definidas **exclusivamente neste nível**.

## Como escrever a Descrição

A Descrição tem **2–3 frases** em linguagem de negócio. Comece pela **capacidade que o
conjunto entrega ao usuário** — o Feature Set *agrupa operações*: **Reúne as operações
de…**, **Agrupa as funcionalidades que permitem…**, **Permite ao [perfil] …** (verbos
diferentes do N1, que *concentra/governa* uma área). Ordene as frases assim: (1ª) quais
operações reúne, já antecipando as features; (2ª) o que isso permite ao perfil. A linha
**Não faz:** delimita o escopo negativo do conjunto.

> **Exemplo — Feature Set Cadastro de Clientes (`CLI-CAD`)**
>
> Reúne as operações de manutenção do cadastro de clientes: incluir, pesquisar, editar,
> visualizar e desativar um cliente. Permite ao operador manter os dados cadastrais sempre
> atualizados a partir de uma única área do sistema.
>
> **Não faz:** análise de crédito, faturamento ou histórico de compras do cliente.

## Artefato

```text
modules/[dominio]/[feature-set]/README.md
```

## Como gerar

O N2 é **integralmente negocial** — passada única (não há passada técnica):

| Prompt | Produz |
|---|---|
| `PROMPT_2A_N2_negocio` | jornada, features, telas e permissões |

> O detalhe técnico do que o Feature Set agrupa vive nos N3 (`PROMPT_3B`).

> Caminho bottom-up: a partir de várias features (N3) é possível **derivar** o N2
> com `PROMPT_N3_TO_N2`.
